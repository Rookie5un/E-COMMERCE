## 2. 产品背景

随着电商平台评论数据持续增长，用户评论已经成为反映商品质量、服务体验和用户需求的重要数据资产。传统人工阅读评论的方式存在效率低、覆盖不全、主观性强、难以量化的问题，无法帮助商家快速定位产品优点、用户痛点和改进方向。

本产品希望通过自然语言处理技术，将海量非结构化评论转化为可查询、可视化、可沉淀的结构化分析结果，帮助商家从评论中识别用户情感倾向、商品功能点表现和高频负面问题，并自动生成可用于决策的分析报告。

## 3. 产品定位

本产品是一个面向电商评论数据的多维分析与可视化平台。平台围绕“评论采集/导入 - 文本预处理 - NLP 分析 - 可视化展示 - 决策报告”形成闭环，为商家提供商品评论洞察能力。

核心价值包括：

- 快速识别整体口碑：对评论进行正向、中性、负向三分类，并展示情感占比与趋势。
- 定位关键功能表现：提取评论中的商品功能点，如屏幕、续航、物流、服务、散热、拍照等，并统计各功能点的提及频次和情感分布。
- 发现核心负面问题：对负面评论进行关键词聚合，识别卡顿、续航短、发热、做工差等高频痛点。
- 降低分析成本：通过可视化看板和 PDF 报告，减少人工阅读评论和整理结论的时间。

## 4. 目标与成功指标

### 4.1 产品目标

- 支持导入或采集电商评论数据，并完成清洗、去重和结构化存储。
- 支持对评论进行情感分析、功能点提取和负面问题挖掘。
- 支持按商品、平台、品类、时间区间查看评论分析结果。
- 支持用图表展示情感趋势、功能点词云、问题排行榜和评论明细。
- 支持自动生成结构化分析报告，供商家进行产品优化和营销决策。

### 4.2 成功指标

| 指标类型 | 指标 |
| --- | --- |
| 分析效率 | 单个商品评论导入后，可在可接受时间内完成批量分析并展示结果 |
| 情感分析效果 | 情感三分类结果可区分正向、中性、负向，并支持人工抽样校验 |
| 功能点识别效果 | 能识别常见电商品类功能点，如价格、质量、物流、服务、续航、屏幕等 |
| 可视化完整度 | 至少包含情感分布、情感趋势、功能点排名、功能点词云、负面问题排行榜 |
| 决策可用性 | 报告中包含商品概览、主要优点、主要问题、优化建议和评论样例 |

## 5. 用户角色与使用场景

### 5.1 角色建议

建议 MVP 阶段只定义一个核心角色：商品运营分析员。该角色可以覆盖“上传评论数据、启动分析任务、查看可视化结果、定位负面问题、导出报告”的完整闭环，不需要在初版中区分商家、运营、客服、数据分析师等多个权限角色。

这样设计的好处是系统边界更清晰，页面流程更短，毕业设计演示也更容易聚焦在 NLP 分析与可视化能力本身。其他人员可以作为报告阅读者或业务干系人存在，但不需要在系统内单独建角色。

### 5.2 核心角色

| 用户角色 | 核心诉求 | 典型场景 |
| --- | --- | --- |
| 商品运营分析员 | 将电商评论转化为可视化分析结果和优化建议 | 导入某商品评论数据，启动 NLP 分析，查看情感分布、功能点表现和负面问题，并导出分析报告 |

## 6. 需求范围

### 6.1 MVP 范围

- 评论数据导入：支持 CSV 文件导入，字段至少包含商品/品类、评论内容、可选情感标签。
- 数据预处理：支持去重、空值过滤、文本清洗、分词、停用词过滤。
- 情感分析：基于 BERT 微调模型或可替换分类模型，实现正向、中性、负向三分类。
- 功能点提取：基于领域词典与 BiLSTM-CRF 或可替代序列标注方案，识别评论中的商品属性/功能点。
- 问题挖掘：对负向评论使用 TextRank 提取高频问题关键词。
- 可视化看板：使用 Vue.js 与 ECharts 展示核心分析图表。
- 报告生成：支持生成商品维度的结构化分析报告，可导出 PDF。

### 6.2 V1.0 可扩展范围

- 电商平台评论采集：支持对京东、拼多多等平台进行合规采集或半自动导入。
- 多平台对比：支持按平台比较商品评论口碑差异。
- 时间趋势分析：支持按天、周、月展示评论数量和情感变化趋势。
- 分析任务管理：支持查看任务状态、失败原因和历史结果。
- 人工校正：支持人工修改情感标签或功能点，用于后续模型迭代。

### 6.3 暂不纳入范围

- 不做交易、订单、支付等电商经营闭环能力。
- 不采集用户隐私信息，不展示个人敏感数据。
- 不在 MVP 阶段实现复杂消费行为预测，如购买转化率、交易价值预测。
- 不承诺对所有电商平台页面结构长期稳定适配，采集能力需遵守平台规则与法律合规要求。

## 7. 核心业务流程

### 7.1 数据流转流程

1. 用户创建商品或选择已有商品。
2. 用户导入 CSV 评论文件，或启动合规评论采集任务。
3. 系统校验字段格式、编码、数据量和空值情况。
4. 系统对评论进行清洗、去重、分词和停用词过滤。
5. 清洗后的评论写入评论数据表，并保留原始文本和处理后文本。
6. 系统生成分析任务并进入 NLP 处理流程。

### 7.2 NLP 分析流程

1. 系统从数据库读取待分析评论。
2. 情感分析模块输出评论情感标签与情感得分。
3. 功能点提取模块识别评论中的商品属性/功能点。
4. 系统将情感结果与功能点进行关联统计。
5. 若评论为负向评论，则进入 TextRank 问题关键词提取。
6. 系统聚合功能点频次、正负向比例、高频问题关键词和关联评论。
7. 分析结果写入情感结果表、功能点提及表、问题聚类表和报告结果表。

### 7.3 可视化与报告流程

1. 用户在前端选择商品、平台、品类和时间区间。
2. 前端请求 Flask 后端分析 API。
3. 后端返回聚合统计数据和评论明细数据。
4. 前端使用 ECharts 动态渲染图表。
5. 用户查看分析结论，可按功能点或问题关键词下钻查看原始评论。
6. 用户点击生成报告，系统输出结构化 PDF 分析报告。

## 10. 数据需求

### 10.1 设计建议

数据库不建议只按“评论表、功能点表、问题表、商品表”来拆分，因为功能点频次和问题次数本质上是分析结果或聚合结果，直接作为主表容易导致后续重跑模型、切换时间范围、对比模型版本时难以追溯。

建议改成“原始数据 - 处理后文本 - 分析任务 - 模型结果 - 报告产物”的结构。这样同一批评论可以被不同模型重复分析，每次分析结果都能关联到具体任务和模型版本，也方便后续做人工校正、结果对比和报告复现。

### 10.2 核心数据表

| 表名 | 关键字段 | 说明 |
| --- | --- | --- |
| products | id, name, category, platform, url, created_at | 商品基础信息。若导入数据集没有明确商品，可用数据集名称或品类作为临时商品 |
| review_batches | id, product_id, source_type, file_name, row_count, imported_count, duplicate_count, failed_count, status, created_at | 评论导入或采集批次，用于记录一次 CSV 导入或一次采集任务 |
| reviews | id, product_id, batch_id, external_id, raw_content, cleaned_content, content_hash, rating, review_time, is_valid, duplicate_of, created_at | 评论主表，保留原文、清洗文本、去重标识和来源信息 |
| analysis_runs | id, product_id, batch_id, status, model_name, model_version, config_json, started_at, finished_at, error_message | 分析任务表，记录一次 NLP 分析的执行状态和模型配置 |
| review_sentiments | id, run_id, review_id, label, confidence, positive_prob, neutral_prob, negative_prob, created_at | 评论级情感分析结果，一条评论在一次分析任务下对应一条结果 |
| aspect_mentions | id, run_id, review_id, aspect_name, normalized_aspect, start_offset, end_offset, confidence, linked_sentiment | 功能点提及结果，记录评论中识别到的属性词及其位置和关联情感 |
| issue_topics | id, run_id, keyword, normalized_keyword, score, frequency, representative_review_id | 负面问题主题或关键词结果，主要由负向评论聚合生成 |
| issue_topic_reviews | id, issue_topic_id, review_id, evidence_text | 问题与评论的关联表，支持点击问题关键词下钻查看原始评论 |
| reports | id, run_id, title, summary_json, pdf_path, created_at | 报告产物表，记录报告摘要结构化数据和 PDF 文件路径 |

### 10.3 关键设计说明

- reviews 只存评论事实和预处理结果，不直接存最终情感标签，避免模型重跑后覆盖历史结果。
- analysis_runs 是结果追溯的核心，所有情感、功能点和问题挖掘结果都应带 run_id。
- review_sentiments 存评论级情感结果，aspect_mentions 存功能点级别结果，两者分开后更容易升级到属性级情感分析。
- issue_topics 不直接存大段评论列表，关联评论通过 issue_topic_reviews 维护，避免字段过长和查询困难。
- reports 只保存报告摘要和文件路径，图表数据可根据 run_id 从结果表重新查询生成。
- 如果 MVP 需要简化，可以先保留 products、review_batches、reviews、analysis_runs、review_sentiments、aspect_mentions、issue_topics 七张表，issue_topic_reviews 和 reports 可在报告导出阶段再补充。

## 11. 算法需求

### 11.1 情感分析

- 输入：单条或批量评论文本。
- 处理：将文本转为 `[CLS] + tokens + [SEP]` 格式，基于 BERT 进行语义编码，在输出层进行 Softmax 三分类。
- 输出：正向、中性、负向标签，以及对应概率或得分。
- 约束：当模型不可用时，可使用已训练的轻量分类器或规则策略作为降级方案，但需要在报告中标识分析方式。

### 11.2 功能点提取

- 输入：评论文本及分词结果。
- 处理：使用 BiLSTM-CRF 或替代序列标注方案识别功能点实体，并结合品类领域词典增强识别。
- 输出：功能点名称、出现位置、关联评论 ID。
- 约束：同义词需要尽量归一化，例如“电池”“续航”“耗电”可根据品类词典映射到同一维度或相关维度。

### 11.3 问题挖掘

- 输入：负向评论集合。
- 处理：对评论分词并构建词共现图，使用 TextRank 计算关键词权重。
- 输出：Top N 问题关键词、出现次数、关联评论列表。
- 约束：需要过滤无业务含义的高频词，避免“东西”“感觉”等词进入问题榜。

## 13. 非功能需求

| 类别 | 需求 |
| --- | --- |
| 性能 | 常规数据量下，图表查询接口应满足前端流畅交互；批量 NLP 分析可异步执行 |
| 可用性 | 导入、分析、导出等耗时任务需要显示状态，不应让用户误以为系统卡死 |
| 可维护性 | 数据采集、预处理、情感分析、功能点提取、问题挖掘、可视化展示应模块化设计 |
| 可扩展性 | 后续可替换 NLP 模型，增加新的电商平台、品类词典和图表类型 |
| 安全合规 | 采集评论需遵守平台规则；不采集与展示用户隐私和敏感信息 |
| 可追溯性 | 评论原文、清洗文本、模型版本、任务日志和分析结果需要可回溯 |

## 14. 验收标准

### 14.1 数据导入验收

- 用户可以成功上传 CSV 评论文件。
- 系统可以识别并提示缺失字段、空评论和重复评论。
- 导入完成后，系统展示成功数、失败数和重复数。
- 评论原文和清洗后文本均可在数据库或页面中追踪。

### 14.2 NLP 分析验收

- 系统可以对评论输出正向、中性、负向标签。
- 系统可以统计商品级情感分布和情感趋势。
- 系统可以识别评论中的功能点，并展示功能点频次。
- 系统可以对负向评论提取问题关键词，并展示问题排行榜。
- 用户可以从功能点或问题关键词下钻查看关联评论。

### 14.3 可视化验收

- 看板至少包含情感分布图、情感趋势图、功能点 Top 榜、负面问题 Top 榜和评论明细表。
- 用户切换商品、平台、品类或时间区间后，图表数据能够刷新。
- 图表展示的数据与后端统计结果一致。

### 14.4 报告验收

- 用户可以为指定商品生成分析报告。
- 报告包含商品概览、评论概览、情感分布、功能点表现、负面问题和优化建议。
- 报告可以导出为 PDF。

## 16. 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 平台采集规则变化 | 评论采集失败或不稳定 | MVP 优先支持 CSV 导入，采集作为可扩展能力 |
| 评论文本噪声较多 | 影响情感分析和关键词提取效果 | 增强清洗规则、停用词词典和领域词典 |
| 细粒度情感关联困难 | 功能点与情感标签匹配不准确 | 先实现评论级情感与功能点共现统计，再逐步升级到属性级情感分析 |
| 模型训练成本较高 | 开发周期变长 | 使用预训练模型、公开数据集或轻量模型进行 MVP 验证 |
| 图表数据计算较慢 | 前端体验下降 | 对聚合结果进行缓存，耗时任务异步执行 |
| 报告结论过于模板化 | 决策价值不足 | 报告中加入真实数据指标、Top 问题和代表性评论样例 |

## 17. MVP 优先级建议

第一优先级应完成“CSV 导入 - 清洗 - 情感分析 - 功能点统计 - 问题关键词 - 可视化看板 - PDF 报告”的闭环。这样即使暂不接入真实平台爬虫，也可以基于公开数据集或本地评论数据完成毕业设计系统演示。

第二优先级再补充合规采集任务、人工校正、模型版本管理、多平台对比和更细粒度的属性级情感关联能力。
